NON-NEGOTIABLE SECURITY REQUIREMENTS:

1. INFRASTRUCTURE DECOUPLING: Do not leak Stytch SDK data models directly into your core service
   logic. Create an 'IdentityProvider' interface inside your domain space. Wrap the concrete Stytch
   client inside a dedicated adapter that handles data transformations.

2. BACKEND IST ISOLATION: The raw Intermediate Session Token (IST) returned from Stytch must never
   be exposed to the browser. It must be trapped in Redis on the backend using a namespaced key of
   the form "ist:{env}:{uuid_v4}" with a 10-minute TTL. The IST key must be DELETED immediately
   after a successful read in step 9 (one-time use — do not rely solely on TTL expiry). The
   frontend must handle only a secure cryptographically random reference ID string.

3. MFA TAMPER PREVENTION: During step 11, you must explicitly parse the Stytch response property
   'MemberAuthenticated'. If this flag is false (indicating that the organization enforces
   Multi-Factor Authentication), you must immediately drop the flow, block session creation, and
   return a clean HTTP 401 response indicating that an MFA step-up verification challenge is
   required.

4. OPAQUE SESSION COOKIE: Do not return Stytch JWT tokens to the browser. Generate an opaque
   session token string using 'crypto/rand'. Drop this token into an HTTP response cookie matching
   these production security flags:
     Name     = 'somo_sid'
     HttpOnly = true
     Secure   = true
     SameSite = Lax
     Path     = "/"
     Domain   = injected from Config struct (do not hardcode)
     Max-Age  = 30 days (2592000 seconds)
   The generated token must also be persisted server-side in Redis (TTL matching Max-Age) keyed as
   "session:{token}". This is required for validation on every subsequent request. There is no
   valid session without a server-side record.

5. CONTEXT HARNESSING: Extract the generated device fingerprint context located in
   'c.Locals("device_fingerprint")' created by our global security pipeline and associate it with
   your internal session data registration.

6. SESSION VALIDATION ENDPOINT: Implement GET /api/auth/me. This endpoint reads the 'somo_sid'
   cookie, looks up the token in Redis, resolves the associated tenant_id and user_id from the
   sessions table in Postgres, and returns a safe user context payload. This is the handoff
   contract for all downstream API middleware. Return HTTP 401 with a clean error body if the
   token is missing, expired, or not found.

7. LOGOUT ENDPOINT: Implement DELETE /api/auth/session (or POST /api/auth/logout). This endpoint
   reads the 'somo_sid' cookie, deletes the token from Redis and the sessions table, then clears
   the cookie by setting Max-Age=0. Return HTTP 204 on success.

IDEMPOTENCY & CONSISTENCY REQUIREMENTS:

8. REGISTER IDEMPOTENCY: Before calling Organizations.Create() in step 10, check whether a tenant
   already exists in Postgres with the given stytch_org_id or school_name (for the authenticated
   identity). If one exists, skip org creation and proceed to session issuance. This prevents
   duplicate Stytch org provisioning on frontend retries or network timeouts. Use the session_ref
   as a natural idempotency scope.

9. TRANSACTIONAL POSTGRES WRITES: Steps 10–12 span an external Stytch call followed by a Postgres
   write. Wrap all Postgres mutations in step 12 (insert tenant, insert user, insert session) in a
   single database transaction. If any row fails to insert, roll back the entire transaction. Since
   Stytch has no two-phase commit, if Postgres fails after a successful Organizations.Create(),
   log a structured WARN-level alert containing the stytch_org_id for manual reconciliation. Do
   not silently swallow this inconsistency.

    ensures request cancellation and timeout propagation work correctly across Stytch API calls,

INPUT VALIDATION REQUIREMENTS:

    'Validate() error' method called at the top of the service Register function before any I/O.

    Return a typed validation error (see domain error taxonomy below) so the handler can map it

DOMAIN ERROR TAXONOMY:

14. Define the following sentinel/typed errors in 'domain.go'. The handler must use a type switch
    or errors.Is check to map each to the correct HTTP status and JSON body:

      ErrInvalidInput      → HTTP 422  { "error": "invalid_input",      "message": "..." }
      ErrExpiredToken      → HTTP 401  { "error": "expired_token",       "message": "..." }
      ErrMFARequired       → HTTP 401  { "error": "mfa_required",        "message": "..." }
      ErrOrgAlreadyExists  → HTTP 409  { "error": "org_already_exists",  "message": "..." }
      ErrNotFound          → HTTP 404  { "error": "not_found",           "message": "..." }
      ErrInternal          → HTTP 500  { "error": "internal_error",      "message": "..." }

    All errors from external calls (Stytch, Redis, Postgres) that do not map to a domain error
    should be wrapped and returned as ErrInternal. Never leak raw SDK error strings to the client.

---

FX LIFECYCLE REQUIREMENTS:

15. LIFECYCLE HOOKS: Any constructor that holds a long-lived connection (Redis client, Postgres
    pool) must register fx.Hook callbacks:
      OnStart: ping the connection and return an error if unreachable (fail fast at boot)
      OnStop:  gracefully close the connection / drain the pool
    This prevents the application from starting in a broken state and ensures clean shutdown.
