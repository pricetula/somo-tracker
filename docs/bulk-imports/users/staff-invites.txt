ROLE & CONTEXT:
You are an expert full-stack engineer and software architect specializing in Go (Go-Fiber, Asynq), PostgreSQL, Redis, Next.js, IndexedDB, RxJS, libphonenumber-js, and TanStack Table/Query/Virtual. You write secure, minimalist, highly performant, production-ready code.

TASK:
Implement the complete front-end component architecture and back-end async worker pipeline for a "Bulk Staff Invitation Utility" within the system.

TARGET BUSINESS LOGIC & ENUMS:
- Allowed target roles for this utility: 'SCHOOL_ADMIN', 'NURSE', 'FINANCE'.
- Role is NOT a per-row field. The component receives a `role` prop from its host page (e.g. `/nurses/add` -> role=NURSE, `/finance/add` -> role=FINANCE) and applies it uniformly to every invitee in the session. The backend must still defensively validate the supplied role against the three allowed enum values on every request.
- tenant_id and school_id are likewise fixed for the entire import session by route/session context, not supplied per row in the uploaded file. (Authorization/tenant-scope enforcement itself is handled outside this spec by a separate coding agent — do not implement it here, but do not trust unvalidated client values either; assume the enforcement layer wraps this code.)
- invitation_status enum and invitations table are created fresh (database was reset — no migration needed, the additions below are baked directly into the initial DDL rather than added via ALTER). Full schema lives in DATABASE SCHEMA section below; the invitations table includes three columns beyond the original draft (import_job_id, error_message, attempt_count) and the enum includes 'invite_failed' beyond the original four values, to represent rows persisted successfully in Stage 1 whose Stytch invite send failed permanently in Stage 2.

- A bulk import never writes to the `users` table directly. `users` rows are created later by the existing accept-invitation flow, which is out of scope here. Stage 1 of this pipeline writes to `invitations` only.

SYSTEM ARCHITECTURE REQUIREMENTS:

1. FRONTEND DATA ACQUISITION & PERSISTENCE (Next.js, IndexedDB, Web Worker, xlsx)
- Provide a UI Dialog component with two primary toggle entry branches:
  * Option 1: Manual Multi-Add Interface (dynamic inline repeating form rows).
  * Option 2: Bulk Import File Upload (drag-and-drop zone).
- For Option 2, determine CSV vs Excel (.xlsx) programmatically from file extension/MIME type. No manual file-type selector.
- File parsing (CSV/XLSX) must run inside a Web Worker, never on the main thread — large files must not block the UI. Parsed rows stream into local persistence incrementally as the worker produces them, not in one blocking pass.
- Local persistence uses IndexedDB directly (e.g. via a thin wrapper such as idb-keyval). Namespace the local collection by tenant_id + user_id + role/page context so a shared/kiosk browser never resurfaces one admin's draft to another admin's session. Apply a TTL (e.g. 24–48h) to drafts; stale drafts past the TTL are silently cleared rather than offered for resume.
- On Dialog init, check IndexedDB for an existing, non-expired draft for the current namespace; if found, prompt the user to resume or clear it.
- Enforce a maximum row count per import (e.g. 5,000); reject larger files client-side with a clear message before parsing begins.

2. FRONTEND VALIDATION, AUTO-CORRECTION & VIRTUALIZATION (TanStack Virtual, RxJS)
- Use a TanStack Virtualized List to render rows from IndexedDB without freezing the DOM at scale.
- Critical errors (block submission, surfaced first, inline-editable cells, writes back to IndexedDB):
  * Missing first_name or last_name.
  * Invalid '@' structural email layout.
  * Duplicate email within the uploaded batch itself — since the whole batch shares one school_id, this is a flat email-level check (case-insensitive); flag every occurrence after the first.
- Warnings & auto-corrections (non-blocking):
  * phone_number: normalize using libphonenumber-js with default country 'KE' (not hand-rolled regex) into E.164. If unparseable or contains non-numeric junk (e.g. "N/A"), set to NULL and surface as a warning rather than silently dropping it. Visually mark any cell that was auto-corrected (e.g. a subtle highlight) so the change is never invisible to the admin reviewing the grid.
- Once critical errors hit 0, hide the main virtualized grid and show a non-editable 5-row sample as a final sanity check before submit.
- On Submit: dispatch the clean payload (including a client-generated temporary row UUID per record, used by the backend to reconcile Stage 1 results — see below) to the Go backend API. On success, purge the IndexedDB collection, reset form state, and close the Dialog.

3. BACKEND ASYNC INGESTION ENGINE (Go, Asynq, PostgreSQL, Redis)
- On receiving the payload: insert a row into `import_jobs` (tenant_id, school_id, role, created_by, total_records), set status='pending', enqueue an Asynq task, return HTTP 202 immediately.
- Frontend hooks into `/api/v1/imports/staff/track/:id` (SSE) via RxJS. Progress events are published over a Redis pub/sub channel keyed by import_job_id, not held only in-process memory — this is required so any backend instance can serve the SSE connection regardless of which instance's Asynq worker is processing the job (horizontal scaling). If the SSE connection drops, the client must fall back to polling `import_jobs.status / processed_records` rather than losing progress entirely.
- The Asynq worker chunks the array into micro-batches of max 200 records.
- STAGE 1 (bulk DB ingestion, per micro-batch):
  * Build one multi-row INSERT for the whole batch (not a per-row loop):
    INSERT INTO invitations (tenant_id, school_id, email, role, invited_by, token, expires_at, first_name, last_name, phone, registration_number, import_job_id)
    VALUES (...), (...), ...
    ON CONFLICT (tenant_id, school_id, email) WHERE status NOT IN ('expired','revoked') DO NOTHING
    RETURNING id, email;
  * Normalize email to lowercase before insert so conflict matching is reliable.
  * Reconcile RETURNING rows back to the original payload rows by the client-generated temporary row UUID (carried through as application-side mapping, not a DB column) rather than by string-matching email.
  * Rows returned = successfully_persisted_users, passed to Stage 2.
  * Rows present in the batch but absent from RETURNING = already-invited duplicates. These do not need a persisted failure record — increment the job's failed_count with reason "duplicate" for the summary, log it, and move on.
  * Genuine structural failures during this insert (FK violation, dropped connection — should be rare, since validation already happened client-side) are caught, the row's raw payload + error written to `import_job_failures`, and the loop continues.
- STAGE 2 (Stytch B2B invitation, per micro-batch):
  * Loop ONLY over successfully_persisted_users, with bounded concurrency (5–10 workers), not fully sequential and not unbounded — respects Stytch rate limits.
  * Idempotency guard: before calling Stytch for a row, skip it if `stytch_member_id` is already set on that row (protects against duplicate invites if Asynq retries the whole task after a partial failure).
  * On success: update the row's `stytch_member_id`.
  * On failure: distinguish transient errors (429/5xx — retry with backoff, up to N attempts within this stage) from permanent errors (4xx other than rate-limit, e.g. blocked domain). Permanent failures set `status = 'invite_failed'` and `error_message`, and increment `attempt_count`.
- After each micro-batch commits, update `import_jobs.processed_records / success_count / failed_count` and publish a progress event over the SSE/Redis channel.

4. POST-IMPORT RECOVERY & RESOLUTION
- On completion, set `import_jobs.status = 'completed'` or `'completed_with_errors'`, broadcast an `import_finished` SSE event, and trigger a persistent in-app notification (scoped to the tenant/admin who triggered the import) showing success count and skipped/failed count.
- Clicking the notification queries `invitations WHERE import_job_id = $1 AND status = 'invite_failed'` directly (no separate dead-letter table for this path — the invitations table already is the ledger) and re-hydrates those rows into the editing grid for correction.
- Resubmitting corrected rows creates a NEW `import_jobs` row with `parent_import_job_id` pointing at the original job, for traceability, and re-runs the same two-stage pipeline. Corrections target the existing invitation row by id (update in place), not a fresh insert — if the email itself was edited, re-validate against the partial unique index before retrying Stage 2.
- Define a retention window (e.g. 90 days) for `import_job_failures` rows and stale `invite_failed` invitations, after which they are purged or archived — this table holds PII (names, emails, phone numbers). Recovery UI and notifications are restricted to admins on the originating tenant.

5. DATABASE SCHEMA (fresh install — database was reset, no migration path needed; create in this order since FKs depend on it)

CREATE TYPE invitation_status AS ENUM ('pending', 'accepted', 'expired', 'revoked', 'invite_failed');

CREATE TABLE import_jobs (
    id                   UUID        PRIMARY KEY DEFAULT gen_random_uuid(),
    tenant_id            UUID        NOT NULL,
    school_id            UUID        NOT NULL,
    role                 user_role   NOT NULL,
    created_by           UUID        REFERENCES users(id) ON DELETE SET NULL,
    status               TEXT        NOT NULL DEFAULT 'pending', -- pending | processing | completed | completed_with_errors
    total_records        INT         NOT NULL DEFAULT 0,
    processed_records    INT         NOT NULL DEFAULT 0,
    success_count        INT         NOT NULL DEFAULT 0,
    failed_count         INT         NOT NULL DEFAULT 0,
    parent_import_job_id UUID        NULL REFERENCES import_jobs(id),
    created_at           TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    started_at           TIMESTAMPTZ NULL,
    completed_at         TIMESTAMPTZ NULL
);

CREATE TABLE IF NOT EXISTS invitations (
    id                  UUID              PRIMARY KEY DEFAULT gen_random_uuid(),
    tenant_id           UUID              NOT NULL REFERENCES tenants(id) ON DELETE CASCADE,
    school_id           UUID              NOT NULL,
    email               VARCHAR(255)      NOT NULL,
    role                user_role         NOT NULL,
    status              invitation_status NOT NULL DEFAULT 'pending',
    invited_by          UUID              REFERENCES users(id) ON DELETE SET NULL,
    token               TEXT              NOT NULL,
    expires_at          TIMESTAMPTZ       NOT NULL,
    accepted_at         TIMESTAMPTZ       NULL,
    first_name          VARCHAR(255)      NULL,
    last_name           VARCHAR(255)      NULL,
    phone               VARCHAR(50)       NULL,
    registration_number VARCHAR(100)      NULL,
    stytch_member_id    VARCHAR(255)      NULL,
    import_job_id       UUID              NULL REFERENCES import_jobs(id) ON DELETE SET NULL,
    error_message       TEXT              NULL,
    attempt_count       INT               NOT NULL DEFAULT 0,
    created_at          TIMESTAMPTZ       NOT NULL DEFAULT NOW(),

    CONSTRAINT fk_invitations_tenant_school FOREIGN KEY (tenant_id, school_id) REFERENCES cbc_schools(tenant_id, id) ON DELETE CASCADE
);

CREATE UNIQUE INDEX uq_invitations_active_email
  ON invitations (tenant_id, school_id, email)
  WHERE status NOT IN ('expired', 'revoked');

CREATE TABLE import_job_failures (
    id             BIGSERIAL   PRIMARY KEY,
    import_job_id  UUID        NOT NULL REFERENCES import_jobs(id) ON DELETE CASCADE,
    raw_payload    JSONB       NOT NULL,
    error_message  TEXT        NOT NULL,
    created_at     TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
