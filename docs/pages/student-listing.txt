1. OVERVIEW & DESIGN PHILOSOPHY
--------------------------------------------------------------------------------
The student listing module must embody a hyper-minimalist approach. Visual noise 
is treated as technical debt. Traditional cards, deep nested container boxes, and 
heavy vertical grids are completely eliminated. The page relies entirely on 
whitespace, typography, and crisp visual states to establish hierarchy. 

Performance metrics require an instantaneous feel even when handling 10,000+ 
student records. This is achieved via DOM virtualization and a streaming, asynchronous 
ingestion pipeline for bulk uploads without relying on heavy queue systems like Asynq.

2. COMPONENT ARCHITECTURE & UI STATES

   A. THE ACTIVE LISTING (Virtual Table Canvas)
      - Structure: Full-width or max-w-7xl canvas without card wrappers. 
      - Header: `text-2xl font-semibold tracking-tight` with an inline muted counter 
        badge indicating total records.
      - Controls Row: Single unified row containing a borderless search box with an 
        inline icon, filter icon button, and a split primary button: [ Add Student | v ].
      - Table Engine: Built via TanStack Table + TanStack Virtual. 
      - Sticky Elements: Table headers (`<TableHeader>`) use `sticky top-0 bg-background/95 
        backdrop-blur-sm z-10`. Horizontal dividers use ultra-low contrast lines 
        (`border-b border-border/40`). Zero vertical borders.
      - Row Optimization: Rigidly fixed row heights (`h-12`) to prevent virtualization 
        layout jitter. High-speed scrolling renders precise layout skeletons matching 
        row metrics exactly.
      - Row Actions: Hidden by default. Triggered via a single ghost ellipsis button (`...`) 
        initiating a shadcn dropdown menu.

   B. THE EMPTY STATE
      - Visuals: Absolute minimalist alignment centered in the viewport. No clip-art graphics.
      - Typography: Header (`text-sm font-medium`), subtext (`text-xs text-muted-foreground`).
      - Call to Action: Minimalist, secondary-styled dual buttons: `[ Manually Add ]` and `[ Upload CSV ]`.

   C. MANUAL ADDITION (Context Preservation)
      - Component: shadcn Sheet/Drawer sliding out from the right edge.
      - Execution: Form stack inside the drawer (`space-y-4`). Retains underlying view state.
      - State Sync: Saving invalidates the active TanStack Query cache. The drawer closes, 
        and the new entry transitions into the virtualized viewport with a brief spotlight accent.

   D. MASS CSV UPLOAD (1,000+ Records Pipeline)
      - Dropzone: Accessible modal containing a dashed, muted boundary drop area. Includes 
        a micro-link to "Download standard CSV template".
      - Validation Block: Client-side file-type confirmation and structural checking.
      - Ingestion Engine: Handled asynchronously via a background Go Goroutine (no Asynq queue layer).
      - Status Interface: The modal collapses down into a miniature status widget in the bottom-right 
        corner (Google Drive upload style) freeing up main screen navigation.
      - Progress Tracking: Real-time telemetry via Server-Sent Events (SSE). Uses standard HTML5 
        `EventSource` API connecting to a streaming endpoint. Includes a linear shadcn `<Progress />` bar.
      - Error Processing: Unsuccessful row iterations append to a temporary storage log (stored in Redis 
        with a 30-minute expiration). On SSE stream termination, an error download link is surfaced.

3. DATA ARCHITECTURE & NETWORK LIFECYCLE (NO-QUEUE APPROACH)
--------------------------------------------------------------------------------
To maintain an lean architectural blueprint, raw Go Goroutines handle concurrent ingestion:

   1. Frontend dispatches CSV payload to `/api/v1/students/import` via multipart form data.
   2. Go-Fiber endpoint reads headers, parses initial schema layout, and spins up a background 
      Goroutine to process data chunks. 
   3. The API instantly sends a HTTP `202 Accepted` response back, delivering an `import_id`.
   4. Frontend dynamically provisions an SSE connection to `/api/v1/students/import/stream?id={import_id}`.
   5. Go backend validates each row sequentially or concurrently. Progress increments are 
      transmitted down the SSE pipeline:
      `data: {"status": "processing", "current": 450, "total": 1240}`
   6. Invalidation or failures map into a collection array. If validation fails for rows:
      - Backend formats a derived error CSV featuring an appended column: `Reason for Failure`.
      - Backend dumps this error file raw string directly into Redis linked via key: `err_log:{import_id}` 
        with an automated TTL set to 1800 seconds.
   7. SSE delivers final payload block:
      `data: {"status": "completed", "success": 1220, "failed": 20, "download_url": "/api/v1/students/import/errors?id=tmp_err_98234"}`
   8. Frontend catches terminal payload, terminates SSE network event connection, and renders the 
      minimalist error recovery control link.

4. END-TO-END QA TEST CASE SCENARIOS
--------------------------------------------------------------------------------

   TEST CASE 1: FRESH INSTALL EMPTY STATE VERIFICATION
   - Objective: Validate interface consistency when zero rows exist in the tenant database.
   - Pre-condition: Authenticated tenant user, student table completely empty.
   - Actions:
     1. Navigate to the student directory endpoint route.
   - Expected Results:
     - Page renders cleanly with zero flash of a broken table framework.
     - Centralized typography prompt accurately surfaces without structural misalignment.
     - Table head rows are hidden or reflect zero base data rows cleanly.
     - Click actions for both `[ Manually Add ]` and `[ Upload CSV ]` fire expected interactive overlays.

   TEST CASE 2: HIGH-SPEED EXPONENTIAL DATA SCROLLING & VIRTUALIZATION LAYOUT
   - Objective: Validate UI response rate, scroll stability, and skeletal placeholder recovery.
   - Pre-condition: Seed 5,000 active student records inside the data store.
   - Actions:
     1. Enter student listing dashboard screen.
     2. Grab scroll thumb-track and pull rapidly down to row marker 3,500+.
   - Expected Results:
     - The browser DOM retains roughly ~30-40 true active DOM nodes matching viewport sizing bounds.
     - Smooth scroll performance tracking above 60fps with zero layout thrashing or browser freeze.
     - Fluid rendering of matching skeleton lines when TanStack Query handles fast data fetching transitions.
     - Sticky table columns header stays pinned to the top of container layout without shifting out of view.

   TEST CASE 3: CONTEXT-PRESERVING MANUAL ENTRY AND POST-SAVE INLINE HYDRATION
   - Objective: Confirm contextual preservation layout while creating manual entries.
   - Pre-condition: Table populated with existing items; layout scrolled to arbitrary index.
   - Actions:
     1. Tap action button [ Add Student ].
     2. Verify shadcn Sheet component renders right-aligned without tearing down current viewport scroll depth.
     3. Populate form input configurations with valid mock telemetry attributes.
     4. Hit save button handler action.
   - Expected Results:
     - Sheet transitions closed smoothly upon validation return code.
     - TanStack Query invalidation updates matching query caches cleanly.
     - The newly added student element is prepended seamlessly to the active array structure.

   TEST CASE 4: PERFECT MASS IMPORT IMMUTABILITY (1,000+ VALID RECORDS)
   - Objective: Ensure smooth background processing of massive, correctly structured data sets.
   - Pre-condition: A valid, uniformly generated CSV document containing 1,500 clean student rows.
   - Actions:
     1. Click "Import CSV", drop the target CSV document directly into the modal drop area zone.
     2. Trigger confirmation action to post ingestion processing pipeline request.
   - Expected Results:
     - Main viewport overlay modal updates cleanly and minimizes down instantly into a micro bottom-right tracking layout widget.
     - Central UI becomes interactive immediately; user can navigate other application dashboards.
     - SSE channel opens smoothly, pushing fractional linear progress metrics updates (e.g. 10% -> 50% -> 100%).
     - Upon processing completion, progress state transforms to absolute success check. 

   TEST CASE 5: PARTIAL BULK INGESTION SYSTEM CORRUPTION & RECOVERY LOOP
   - Objective: Validate operational error log aggregation, Redis transient lifecycle and error recovery download download flow.
   - Pre-condition: Prepare CSV document with 1,050 records total. 1,000 records structured perfectly; 50 rows broken intentionally (e.g., alphanumeric text inside age field, empty guardian names).
   - Actions:
     1. Execute upload sequence on dirty data file template.
     2. Monitor minimized background processing container widget telemetry stream.
   - Expected Results:
     - Status tracker displays incremental metrics correctly up to terminal loop marker index.
     - SSE passes specific metadata layout mapping 1000 successes and 50 payload rejections.
     - UI reveals an explicit error notification containing the download link: `/api/v1/students/import/errors?id=tmp_err_XYZ`.
     - Clicking the action link fetches a newly synthesized tracking file containing exclusively the 50 corrupt data rows with an explicit appended error rationale header context.

   TEST CASE 6: NETWORK DROP RESILIENCY ON PROGRESS LIFECYCLE
   - Objective: Validate frontend SSE auto-recovery handling if the connection is dropped mid-processing.
   - Pre-condition: Upload process running on a large data file (e.g., 5,000 rows).
   - Actions:
     1. Start upload and wait for progress to reach 30%.
     2. Simulating a network drop by toggling browser DevTools "Offline" state for 5 seconds, then toggle back to "Online".
   - Expected Results:
     - The UI progress bar switches to a warning state or indicates a reconnection phase gracefully.
     - Built-in HTML5 `EventSource` automatic retry mechanics fire reconnect requests back to the Go backend.
     - The Go backend identifies the ongoing state of the `import_id` and resumes pushing accurate stream progress ticks matching actual worker status.
     - The process reaches termination state without missing steps or locking up the UI thread.

   TEST CASE 7: MALFORMED SCHEMA ATTRIBUTE AND HEADER BLOCK CRASH INGESTION
   - Objective: Secure data pipeline edge validations against completely wrong files.
   - Pre-condition: Prepare an invalid CSV file (e.g., a file containing random financial headers or a completely different layout schema).
   - Actions:
     1. Drag and drop the invalid file into the system interface container.
   - Expected Results:
     - Client-side or rapid pre-flight server validation identifies missing mandatory headers (e.g., `student_name`) within milliseconds.
     - The file drop interface immediately updates to a clear, desaturated red warning layout.
     - Explains the exact missing constraints or columns explicitly.
     - Zero backend worker routines are spawned, and no data corruptions leak into the Redis or Postgres layer.
