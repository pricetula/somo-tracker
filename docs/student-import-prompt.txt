--- ARCHITECTURAL OVERVIEW & STYLE GUIDE ---
You are tasked with building a hyper-minimalist, high-performance Student Bulk Importation
feature for Somotracker. The system must support two ingestion patterns: Manual Matrix
Ingestion and Multi-Stage Document Mapping Ingestion (CSV/Excel).

Design Framework Constraints:
- UI Stack: Pure minimalist shadcn/ui primitives.
- Visual Noise: ABSOLUTELY NO useless divs, cards, or unnecessary borders. Use space,
  clean typography, and subtle colors to create visual hierarchy.
- Local Storage State: All step changes, mapping selections, and structural edits must
  actively mirror data to an IndexedDB store named `somotracker_import_engine` under an
  object store named `pending_student_imports`. This enables session recovery if a user
  refreshes or closes the tab mid-workflow. See Session Recovery Contract below.
- High Data Volumes: The system must easily scale to process 1,000+ records smoothly
  at 60fps.

--- SESSION RECOVERY CONTRACT ---
On application mount, before rendering any import UI, check for an existing unfinished
session in IndexedDB:
- If a stored session is found, display a non-blocking top banner:
  "You have an unfinished import session from [timestamp]. Resume or start fresh?"
  Provide two actions: [Resume Session] and [Discard & Start New].
- [Resume Session]: Rehydrate wizard state from IndexedDB, restoring the user to their
  last confirmed step with all mapping selections and record data intact.
- [Discard & Start New]: Purge the `pending_student_imports` object store entirely,
  then initialize a clean session state.
- If no stored session exists, initialize silently and proceed directly to the import UI.

--- TARGET BACKEND CONTRACT (MOCK) ---
Assume the frontend will orchestrate a single bulk POST request to:
`POST /api/v1/imports/students`
Payload Content Type: application/json

Expected Body Structure:
[
    {
        "full_name": "string (required)",
        "gender": "M | F (required)",
        "date_of_birth": "string (YYYY-MM-DD, optional/nullable)",
        "upi_number": "string (optional/nullable)",
        "knec_assessment_number": "string (optional/nullable)",
        "cbc_student_parents_id": "string (uuid, optional/nullable)",
        "class_id": "string (uuid, optional/nullable — overrides root class_id if present)"
    }
]


--- BULK POST RESILIENCE CONTRACT ---
- Set a client-side timeout of 30 seconds on the POST request.
- On network failure or timeout: display a non-blocking error toast with a [Retry] action.
  Do not clear the staged IndexedDB data. Allow the user to retry without re-uploading.
- On HTTP 207 Multi-Status (partial success): Parse the response body for per-row
  success/failure indicators. Display a results summary screen listing failed rows with
  server-provided reasons. Allow the user to correct and resubmit only the failed subset.
- On HTTP 4xx/5xx: Display a descriptive error state with the server message if available.
  Preserve staged data for retry.
- On full success (HTTP 200/201): Display a success confirmation screen with total records
  imported. Purge IndexedDB session data after confirmed success.

PHASE 1: LOOKUP SYNCHRONIZATION DATA LAYER
================================================================================
Before any importation screen or mapping wizard initializes, execute parallel background
queries to build fast-lookup reference mappings from the backend.

--- LOOKUP FAILURE HANDLING ---
Each lookup (parents, classes) must be treated independently:
- If a lookup request fails (network error, non-2xx response, timeout): display a
  non-blocking warning banner specific to that lookup:
  e.g., "Parent records could not be loaded. Parent linking will be unavailable."
- The import workflow must NOT be blocked. Proceed with degraded mode:
  * Failed parents lookup → all parent fields resolve to null. A persistent inline notice
    must appear on the parent mapping step: "Parent linking unavailable — lookup failed."
  * Failed classes lookup → all class fields resolve to null. A persistent inline notice
    must appear on the class mapping step: "Class linking unavailable — lookup failed."
- Provide a [Retry Lookup] action on each warning banner. Retrying a successful lookup
  replaces the degraded null behavior without restarting the wizard.

1. Parents Reference Mapping:
   - Query a list of existing school parents.
   - Map them into a high-performance O(1) JavaScript Map object keying normalized names
     to parent records.
   - Map Key Normalization Formula: `str.toLowerCase().replace(/\s+/g, "")`
   - Example: `full_name: "Nancy Onyinde"` → key `"nancyonyinde"`.

2. Classes Reference Mapping:
   - Query active classes matching the organization's current academic year context.
   - Map records into an O(1) Map object.
   - Formatting Resiliency: Spreadsheets often vary (e.g., "Class 4 West", "Grade 4 West",
     "Std 4 West", "G4 West"). The normalization key generator must be precise.

   REVISED NORMALIZATION APPROACH — DO NOT use a naive global character strip.
   The previous pattern `replace(/(class|grade|std|g|form)/g, "")` is destructive:
   it mangles mid-word characters (e.g., "Geography" → "eography", "Stage" → "Ste").

   Instead, apply prefix-only stripping using a word-boundary-aware pattern:
   ```
   function normalizeClassName(raw) {
     return raw
       .toLowerCase()
       .trim()
       // Strip known instructional prefixes only at the START of the string
       .replace(/^(class|grade|std|form)\s+/, "")
       // Strip single-letter prefix "g" or "g." only at the START, not mid-word
       .replace(/^g\.?\s*/, "")
       // Collapse all remaining whitespace
       .replace(/\s+/g, "");
   }
   ```
   Examples:
   - "Class 4 West"   → "4west"       ✓
   - "Grade 4 West"   → "4west"       ✓
   - "G4 West"        → "4west"       ✓
   - "Std 4 West"     → "4west"       ✓
   - "Form 4 West"    → "4west"       ✓
   - "Grade 10 Geography" → "10geography"  ✓ (no mid-word destruction)
   - "Stage 2"        → "stage2"      ✓ (prefix not stripped — not a class prefix)

   Apply this same function when normalizing the extracted class name string from each
   student record during the wizard mapping step.

PHASE 2: INGESTION VECTOR ORCHESTRATION
================================================================================
The landing layout presents a minimalist toggle or split screen choosing between two
ingestion vectors.

--- SHARED DUPLICATE DETECTION (BOTH PATTERNS) ---
Before staging records into IndexedDB, run a client-side duplicate check against
existing system records retrieved during Phase 1 (or a dedicated duplicate-check endpoint
if available). Flag records where:
- `upi_number` matches an existing student UPI (exact match).
- `full_name` + `date_of_birth` combination matches an existing student record
  (normalized name match + exact DOB).

Flagged duplicates must:
- Be rendered in the validation screen (Phase 4) with a distinct amber/warning highlight
  (separate from red validation errors).
- Display the reason: "Possible duplicate: matches existing student [Name]."
- Include a per-row override checkbox: "Import anyway" — allowing intentional re-imports
  (e.g., student transferring between classes).

--- PATTERN A: MANUAL ENTRY MODE ---
1. Render a clean grid/matrix line-item interface using a virtualized row renderer
   (TanStack Virtual) to maintain 60fps performance even when the user has added
   hundreds of rows manually. Do NOT render a plain unvirtualized grid for this pattern.

2. Field column list:
   - Full Name (Input)
   - Gender (shadcn Select restricting values to M/F)
   - Date of Birth (Date Picker)
   - UPI Number (Input)
   - KNEC Assessment Number (Input)
   - Parent (shadcn Combobox pointing to Parents Map lookup)
   - Class (shadcn Combobox pointing to Classes Map lookup)

3. Column Conflict Guard:
   Not applicable to manual mode since each field maps to a dedicated column.
   Conflicts are only possible in Pattern B — see Column Conflict Guard there.

4. "Next" Validation Step:
   - When the user triggers "Next", run a client-side synchronous verification pass
     over all active row states.
   - If missing required fields or format violations exist, do NOT transition pages.
     Display inline red rings around flawed cells.
   - Keep the "Upload" action disabled until the error count reaches zero.
   - If clean, POST the data array to the API endpoint per the Bulk POST Resilience
     Contract above.

--- PATTERN B: COLUMN CONFIGURATION WIZARD (FILE UPLOAD) ---
Upon dropping a CSV or Excel file into a dropzone, parse it using PapaParse with
`worker: true` to avoid blocking the main UI thread.

WORKER DATA TRANSFER NOTE: PapaParse with `worker: true` returns parsed results via
`postMessage`. For files with 1,000+ rows, the serialized payload transferred back from
the worker to the main thread can itself cause a brief UI stall due to structured clone
overhead. To mitigate:
- Transfer only the header array and the first 5 valid rows back immediately for wizard
  preview use.
- Transfer the full record array in a deferred second message after the wizard mapping
  steps are complete, triggered just before Phase 3 processing begins.
- Display a subtle "Parsing file…" progress indicator during the deferred transfer.

Guide the user through a progressive multi-stage layout. Each step must:
- Clearly explain its intent at the top.
- Show a live preview using the first valid data row (skipping blank rows and the
  header row itself).
- Provide step confirmation actions.

--- COLUMN CONFLICT GUARD (PATTERN B) ---
A spreadsheet column must not be mapped to more than one wizard field simultaneously.
- When a user selects a column in any step, mark it as "in use."
- In all other step Select components, visually dim (but do not hide) already-claimed
  columns and append a label: "(mapped to [Field Name])".
- If a user selects a column already claimed by another field, show a non-blocking
  inline warning: "This column is already mapped to [Field Name]. Re-mapping will
  remove it from that step." Provide [Confirm Remap] and [Cancel] actions.

WIZARD STEPS:

1. Student Name Extraction Step:
   - Screen Explainer: Explains intent to properly assemble the full student identity.
   - Preview Slot: Shows a reactive name built from current column selections. If no
     columns are selected, display faint placeholder "John Doe". If a column auto-matched
     as 'full_name' or 'name', render the first valid row value as a preview.
   - Select Array UI: A shadcn Select listing discovered header keys. Selecting a header
     immediately pulls that key's value from the first valid data row as live feedback.
   - "(+) Add Column Component" button adds a second select beneath the first. Multiple
     selections are concatenated using `.trim()` with a single space `" "` between segments.
   - Confirm to advance.

2. Gender Extraction Step:
   - Single shadcn Select to identify the spreadsheet gender column.
   - Confirm to advance.

3. Date of Birth Extraction Step:
   - Single shadcn Select to identify the date of birth column.
   - Includes a clear [Skip Column Mapping] action alongside standard confirmation.

4. UPI Number Extraction Step:
   - Single shadcn Select to target the UPI column.
   - Includes [Skip Column Mapping].

5. KNEC Assessment Number Extraction Step:
   - Single shadcn Select to target the KNEC assignment code column.
   - Includes [Skip Column Mapping].

6. Parent Name Extraction Step:
   - Screen Explainer: Explains intent to bind students to existing parent records.
   - Preview Slot: Reactive value preview from selected columns. Faint "John Doe"
     if no columns chosen.
   - Multi-column concatenation via "(+)" buttons, identical to Name step logic.
   - On confirmation: normalize the compiled string using:
     `key = extractedString.toLowerCase().replace(/\s+/g, "")`
     Look up in the Parents Map. If matched, bind the parent UUID into the row state.
     If unmatched, parent field stays null — no error is raised (parent linkage is
     optional), but a per-row advisory note "Parent not found in system" is surfaced
     in the Phase 4 review screen for operator awareness.
   - Includes [Skip Column Mapping].

7. Class Name Extraction Step:
   - Screen Explainer: Explains intent to allocate students into targeted class streams.
   - Preview Slot: Reactive value preview. Faint "Class 1 North" if no columns chosen.
   - Multi-column concatenation via "(+)" buttons (e.g., merging a "Grade" column and
     a "Stream" column into "Grade 4 West").
   - On confirmation: normalize using the REVISED `normalizeClassName()` function defined
     in Phase 1. Look up in the Classes Map. If matched, bind the class UUID into the
     row state. If unmatched, class_id stays null — no error raised, but a per-row
     advisory note "Class not found in system" is surfaced in Phase 4.
   - Includes [Skip Column Mapping].

PHASE 3: BACKGROUND STAGING & INDEXEDDB MERGE LAYER
================================================================================
When the wizard completes all configuration phases, execute a background processing
script over the full dataset.

FILE SIZE & ROW COUNT LIMITS:
- Maximum file size: 10MB. Reject files exceeding this with a clear error before parsing.
- Maximum row count: 5,000 rows. If the parsed record count exceeds this, display an
  error: "This file contains [N] rows. The maximum supported per import is 5,000.
  Please split the file and import in batches." Do not proceed.
- These limits should be surfaced to the user on the dropzone UI before file selection
  to set expectations.

RECORD STRUCTURE:
Construct rows conforming to this TypeScript interface:

```typescript
interface StagedStudentRecord {
  _rowIndex: number;                  // Original row index from source file/matrix
  full_name: string;
  gender: "M" | "F" | null;
  date_of_birth: string | null;       // YYYY-MM-DD or null
  upi_number: string | null;
  knec_assessment_number: string | null;
  cbc_student_parents_id: string | null;
  class_id: string | null;
  isValid: boolean;
  isDuplicate: boolean;               // From duplicate detection pass
  importAnyway: boolean;              // User override for duplicates
  errors: Record<string, string>;     // Field key → human-readable error message
  advisories: Record<string, string>; // Field key → non-blocking advisory note
}
```

VALIDATION PASS:

* full_name: Must be a non-empty string after trimming. Error if blank.

* gender: Normalize before validating using this explicit mapping table:
  ```
  const GENDER_MAP: Record<string, "M" | "F"> = {
    "m": "M", "male": "M", "boy": "M", "b": "M", "1": "M",
    "f": "F", "female": "F", "girl": "F", "g": "F", "2": "F",
    // Add Swahili equivalents:
    "me": "M",   // mwanaume
    "ke": "F",   // kike
  };
  const normalized = rawGender?.toString().toLowerCase().trim();
  const resolved = GENDER_MAP[normalized] ?? null;
  ```
  If resolved is null, record error: `"gender": "Unrecognized gender value: '[raw value]'"`.
  If resolved is "M" or "F", store the canonical value.

* date_of_birth: If present, validate as a parseable date. Attempt to coerce common
  formats (DD/MM/YYYY, MM/DD/YYYY, D-M-YYYY) to ISO YYYY-MM-DD. If unparseable,
  record error: `"date_of_birth": "Unrecognized date format: '[raw value]'"`. If
  ambiguous (e.g., 01/02/2010 could be Jan 2 or Feb 1), flag as an advisory, not
  an error: `"Ambiguous date — assumed DD/MM/YYYY. Verify if incorrect."` and store
  the DD/MM interpretation.

* upi_number: If present, validate against UPI format rules (define regex per Kenya
  NEMIS specification). Record error if format is violated.

* knec_assessment_number: If present, validate against KNEC format rules. Record
  error if format is violated.

INDEXEDDB WRITE:
- Write the complete structured record array to IndexedDB in a single transaction.
- Do not hold this array in React component state. Read from IndexedDB on demand in
  the Phase 4 view to keep memory pressure low for large datasets.
- Store a session metadata record alongside the student records:
  ```
  {
    sessionId: string (uuid),
    createdAt: ISO timestamp,
    lastUpdatedAt: ISO timestamp,
    currentStep: number,
    totalRecords: number,
    ingestionPattern: "manual" | "csv",
    mappingConfig: { /* wizard selections */ }
  }
  ```

PHASE 4: HIGH-PERFORMANCE VIRTUALIZED VALIDATION MOTOR
================================================================================
Transition the interface into a virtualized list view built with TanStack Table and
TanStack Virtual. This screen manages records with error indicators or format anomalies.

--- VIEW TOGGLE ---
Provide a segmented control at the top of the screen:
  [Errors & Warnings] [All Records]

- "Errors & Warnings" view (default): Shows only records where `isValid === false`
  OR `isDuplicate === true` (and `importAnyway === false`).
- "All Records" view: Shows every staged record, allowing operators to audit the full
  import set before submission.

Both views use the same TanStack Virtual renderer.

1. Layout Rendering Engine:
   - Render editable cells using custom minimalist shadcn input wrappers.
   - Cells with errors draw a dark red ring accent.
   - Cells with advisories (non-blocking) draw a muted amber ring accent.
   - Duplicate rows draw an amber left border on the row level.
   - Resolved rows (isValid: true, cleared for removal) draw a muted green left border.

2. Staged Batch Clear — Anti-Scroll-Hop Behavior:
   When a user satisfies a cell's error validation:
   - Mutate the cell border accent to clean green.
   - Reduce the row's opacity slightly to indicate resolved status.
   - Set `isValid: true` in IndexedDB for that record.
   - KEEP the row visible in the viewport. Do NOT unmount it immediately.
   - Only physically remove resolved rows from the rendered list when the user explicitly
     clicks the top bar command [Clear Resolved Rows], or when final submission is
     triggered (at which point all remaining resolved rows are flushed automatically).

3. Performance Optimization Rules:
   - On cell edit: run validation ONLY on the affected row record. Do not re-validate
     the entire collection.
   - Write the updated row directly to IndexedDB by its `_rowIndex` key.
   - Update only the local row state slice in the virtualized list. Do not trigger a
     full list re-render.

4. Duplicate Row Handling in Validation Screen:
   - Duplicate rows appear with amber styling and the advisory:
     "Possible duplicate: matches existing student [Name]."
   - Each duplicate row has a checkbox: [✓ Import anyway].
   - Checking it sets `importAnyway: true` and removes the row from the error/warning
     count. The row fades to resolved styling.

5. Form Action Execution:
   - Persistent top-aligned control bar tracks:
     * Outstanding error count (red badge)
     * Outstanding duplicate warnings not yet overridden (amber badge)
     * Total records staged (neutral)
   - [Clear Resolved Rows] button: flushes all `isValid: true` rows from the visible
     list without affecting the underlying IndexedDB data (they remain staged for
     submission).
   - Final [Submit Import] action:
     * Disabled while outstanding error count > 0 OR unchecked duplicate warnings > 0.
     * On activation: reads all records from IndexedDB, constructs the final API payload
       per the backend contract, and issues the POST per the Bulk POST Resilience Contract.
     * Excludes duplicate rows where `importAnyway === false`.
     * Includes duplicate rows where `importAnyway === true`.
